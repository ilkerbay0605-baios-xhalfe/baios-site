Project Discipline

Each project is its own world. When BAIOS opens a project, it reads these files:

~/.baios/projects//
├── project_context.md      ← permanent: purpose, rules, roles
├── project_tmp_context.md  ← temporary: recent operations, working notes
├── project_config.json     ← connected ARMs, skills, flows, automations
├── discipline_v.md       ← working method produced with the model
├── docs/                    ← project materials
└── outputs/                 ← skill/flow outputs

Discipline Engine: It combines the project's materials, its goal, and BAIOS's current capabilities, and has the model produce a step-by-step working method. Each version is written to a file and versioned.

When the project changes, the discipline can be regenerated. Old versions stay in the archive.

Context Layers

BAIOS does not send the model a single block of text — it sends it in layers:

LayerContentWhen
SystemBAIOS version, ARM/skill/flow counts, installed ARMs, active automationsEvery message
ProjectActive project info, project_context.md content, connected componentsIf a project is active
Temporaryproject_tmp_context.md — recent operationsIf a project is active
WikiWiki pages relevant to the query (RAG)If a relevant page exists
ConversationLast N messagesEvery message
User messageThe current question/requestEvery message

Why it matters:

  • The model won't make the "BAIOS doesn't have that skill" mistake (the list is in front of it).
  • If the project changes, the context changes on its own.
  • When a new ARM is added, all model conversations know it automatically — no stale list.

ARM — Adapter

ARM = the adapter that connects BAIOS to the outside world.

Categories:

  • API-based: Gmail, GitHub, Telegram, Notion, Brave Search, OpenWeather…
  • Local: shell, file, file_read, web_reader, browser, sql, http, rss
  • Cloud/IoT: cloud (S3/GDrive), MQTT, HomeAssistant
  • System: git_ops, deploy, feedback…
Setup levels (install_level):
  • 0 — No setup, runs directly (file, shell, web_reader)
  • 1 — Needs an API key (brave_search, weather)
  • 2 — OAuth / multi-field setup (gmail, notion, github)
  • 3 — Risky (shell)
Each ARM:
  • Handler Python function: handle_(action, config, query)
  • Catalog entry: baios/arm/catalog.json
  • Version: baios/arm/versions_local.json
  • Setup flow: the fields to ask for (token, repo, etc.)
To write a new ARM, see the developer guide.

Skill — Repeatable Workflow

Skill = a template that runs step by step. A JSON file.

{
  "name": "research",
  "required_arms": ["brave_search"],
  "input_params": ["topic"],
  "steps": [
    {"action": "arm:brave_search", "params": {"query": "{topic}"}},
    {"action": "llm:Summarize the results in 5 bullet points."},
    {"action": "file:outputs/research_{topic}.md"}
  ]
}

Step types:

  • arm: — run an ARM
  • llm: — process with the model (use_previous: also send the previous output)
  • file: — save
  • skill: — call another skill
Parameters: placeholders like {topic}, {audience} are filled at call time.

Skill JSON files can be sent to your friends and added to the community store.

Flow — Large Multi-Skill Flow

A flow runs multiple skills/ARMs with conditions and loops.

Example: the site_release flow (for baios.dev) 

generate_changelog → release_notes → git_ops(branch) → git_ops(commit)
  → git_ops(push) → github(pr_create) → site_qa

Each step can be conditional: "if QA fails, open a rollback branch."

Automation — Time/Trigger

Automation runs a skill/flow on a schedule or by an event.

  • **Cron:** `0 8 * * *` — every morning at 08:00
  • **Webhook:** incoming HTTP → trigger a flow
  • **File change:** watch `~/downloads/`, analyze if a new PDF appears
  • **RSS:** new item → summarize + email
  • **MQTT:** sensor threshold → notification
Every automation is **approved** — rule-based; if the risk is low it runs without re-asking.

MCP — Model Context Protocol

BAIOS is both an MCP server and an MCP client.

  • **Server:** tools like Claude Desktop and VS Code can use BAIOS over MCP — "open a PR in ilker's baios-dev project."
  • **Client:** BAIOS connects to any MCP server (Figma, Jira, Notion API, GitHub MCP, Slack MCP…). Adding an MCP server = adding a new ARM to BAIOS.
Details: [MCP guide](../kaynaklar/mcp-rehberi.html).

Security

  • **Vault:** API keys encrypted with Fernet + PBKDF2. Plaintext fallback only if the vault is locked.
  • **Path traversal protection:** Every file path is resolved and checked to be inside the base dir.
  • **Shell whitelist:** Only whitelisted commands. `rm -rf`, `sudo` are blocked.
  • **git_ops sandbox:** Only permitted repos. `push` opt-in, `--force` opt-in.
  • **Audit log:** Every action recorded under `~/.baios/audit/`. Tamper-proof for enterprise.
  • **Input validation:** Every API endpoint validated with Pydantic.
